Добавлен конструктор в класс Greeter, который присваивает полю formality значение по умолчанию, 
чтобы избежать проблемы с отсутствующим значением при создании экземпляра класса.

Для того чтобы можно было легко добавлять новые виды приветствий, предлагаю вынести кейсы 
из условной конструкции switch в отдельные классы. Таким образом, мы сможем изменять и расширять 
варианты приветствий без изменения кода в классе Greeter, что соответствует принципу открытости/закрытости (OCP).

Вместо использования поля formality предлагаю убрать уровни официальности из класса Greeter. 
Каждый вариант приветствия будет иметь свой собственный класс, который будет реализовывать интерфейс Greeting. 
Это позволит более гибко управлять различными видами приветствий. Также предлагаю добавить конструктор в Greeter, 
который будет устанавливать вид приветствия по умолчанию, если никакой другой вариант не передан.
